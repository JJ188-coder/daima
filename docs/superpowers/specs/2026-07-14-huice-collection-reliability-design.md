# 慧经营与拼多多采集可靠性修复设计

## 背景

2026-07-12、2026-07-13 的实际补采暴露出六类问题：同日商品重采不会移除旧行、成功重跑不会清除旧失败标记、日期选择器可能在不同残留面板中完成两次点击、多个推广页会导致采集对象不确定、慧经营 XLSX 会产生已知的 openpyxl 样式警告，以及下载中心曾受通知弹窗和全局 DOM 查找影响。

本次修复采用最小边界，不重写采集器，也不改变现有利润口径、店铺映射结构或扩展数据格式。

## 目标与成功标准

1. 完整重采某日商品数据时，SQLite 中该日期的数据与最新 XLSX 快照完全一致。
2. 下载失败、解析失败或空快照不得清除现有数据库数据。
3. 只有整次采集完全成功后，旧 `failed-dates.json` 才会被删除。
4. 商品和店铺日期选择器的开始、结束点击必须落在同一个可见目标月份面板中。
5. 多个拼多多推广页存在时，保留 `mallId=338884784` 的“吃了会快乐零食店”页面并关闭另一个推广页。
6. 下载中心操作只依赖下载业务层，不依赖关闭“我知道了”等通知弹窗。
7. 只过滤慧经营 XLSX 已确认无害的 openpyxl 默认样式警告，其他警告保持可见。
8. 所有改动通过自动测试、语法检查和真实 CDP 页面验证。

## 非目标

- 不统一重写商品、店铺、推广三个采集器的完整状态机。
- 不改变 `pdd_shop_mapping` 的数据库结构。
- 不将 `mallId=338884784` 硬编码成新的源代码映射；继续使用已确认的数据库映射。
- 不通过点击全局遮罩或未知弹窗恢复下载。
- 不改变 XLSX 的业务字段解析规则和利润计算规则。

## 设计

### 1. 商品日期快照原子替换

在 `scripts/huice/lib/db.mjs` 新增专用日期快照替换函数，不改变通用 `bulkUpsertProductProfit()` 的语义。

输入为一个已验证、非空、且属于同一日期的商品记录集合。函数在单个 SQLite 事务中：

1. 校验记录集合非空；
2. 校验所有记录日期一致；
3. 删除 `product_profit` 中该日期的全部旧行；
4. 使用现有标准化和 upsert 逻辑写入本次全部记录；
5. 成功提交，异常自动回滚。

`tools/huice-export-cdp.mjs` 只对已经完成下载、通过 XLSX 解析和非零记录验证的日期调用该函数。每个成功日期独立替换，某个失败日期不会影响其他日期，也不会清除该失败日期的旧数据。

这样，2026-07-13 的 686 条旧数据库记录会被最新 665 条完整快照替换，21 条已不存在的旧商品会被移除。

### 2. 失败标记生命周期

商品和店铺导出器统一采用以下规则：

- 存在失败日期：覆盖写入本次最新 `failed-dates.json`；
- 所有请求日期均成功、结果非空、正常走到完成分支：删除已有失败标记；
- 启动失败、CDP 连接失败、零记录、解析异常或流程提前终止：不删除旧失败标记。

删除操作必须位于明确的完全成功分支，不能仅根据 `failedDates.length === 0` 推断成功。

### 3. 日期选择器稳定性

商品和店铺导出器的日期面板逻辑保持现有“同一天点击两次形成单日范围”的方式，但统一面板选择规则：

1. 打开前点击页面空白处，收起已知残留面板；
2. 只考虑可见的日期面板；
3. 根据月份标题定位目标月份；
4. 当存在多个匹配面板时，固定使用最后挂载的可见面板；
5. 第一次和第二次点击都重新查询 DOM，但使用同一面板选择规则；
6. 两次点击都先找 `td.available`，再兼容已经选中而失去该类名的目标日期单元格；
7. 最终以开始、结束输入框实际值为准；不点击任何无关日期清空状态；
8. 验证失败时维持现有刷新重试机制，最多三次。

### 4. 推广标签页选择和关闭

`tools/pdd-promo-cdp.mjs` 将推广目标选择拆为可测试的候选发现、店铺识别和关闭操作：

1. 枚举全部 `yingxiao.pinduoduo.com` 页面；
2. 逐个连接并读取 `mallId`；
3. 保留 `mallId=338884784` 的目标页；
4. 检测到另一个推广页时，通过 CDP Target 关闭该页；
5. 采集前再次验证保留页仍为目标 `mallId`；
6. 若没有目标店铺页，停止并输出清晰错误，不将其他店铺数据写入“吃了会快乐零食店”；
7. 入库继续通过 `pdd_shop_mapping` 已确认映射写入 `shop_id=9`。

本设计按用户确认允许关闭另一个预先存在的推广标签页，但只关闭推广域名页面，不关闭普通拼多多页面或其他浏览器页面。

### 5. 下载中心分层操作

商品和店铺导出器将下载流程划分为三层：

- 路由层：确认页面位于下载中心路由；
- 业务层：以可见 AG-Grid 为锚点，沿祖先链识别其所属的可见下载中心容器，并只在该根节点内访问 `gridApi`、查询按钮、任务行及下载按钮；
- 通知层：单独记录“我知道了”、消息条等可见通知及其层级信息，但不让它们参与业务判断，也不把关闭通知作为下载前置条件。

业务层识别采用“已知容器优先、结构验证兜底”的方式：

1. 在下载中心路由内枚举可见的 `.v-ag-grid`，不直接使用全页面第一个匹配项；
2. 优先选择位于可见 `.analyzerContainer.view` 内、且具有可用 `gridApi` 的表格；
3. 通过表格祖先链、任务数据和操作列验证该根节点确实属于下载中心；
4. 若出现多个候选业务层，必须依据可见性和下载任务结构消歧，不能静默选择第一个；
5. 将识别出的业务根节点贯穿查询、任务匹配、虚拟行展开和下载点击全过程。

具体约束：

- 不使用全局 `document.querySelector('.v-ag-grid')` 直接决定目标下载表格；
- 任务读取优先使用业务层内 AG-Grid 的 Vue `gridApi`，DOM 只作为同一业务层内的回退；
- 查询和下载均通过业务层内精确定位到的元素触发，不搜索全页面第一个同名按钮；
- 即使通知层的 `z-index` 更高，也直接在业务层 DOM 中触发目标任务操作，不先关闭通知；
- 不点击全局遮罩、未知确认按钮或业务对话框；
- 查询按钮不存在时，只要路由、业务根层和 AG-Grid 已就绪，就继续轮询任务；
- 导出提交后的被动通知清理可以保留为收尾动作，但下载中心任务识别和按钮点击不得依赖该清理成功；
- 调试日志可记录业务层祖先链、通知文本和候选按钮，但不得把通知层按钮当作业务按钮。

### 6. XLSX 已知警告处理

商品和店铺的嵌入式 Python 解析器使用 `warnings.filterwarnings()`，只过滤：

```text
Workbook contains no default style, apply openpyxl's default
```

过滤范围限定为 `UserWarning` 和精确消息。其他警告仍输出。读取并物化所需行后显式关闭 workbook。

本次不切换 `read_only=True`，因为该模式不能消除警告，且现有商品解析测试要求普通加载模式。

## 错误处理与数据安全

- 日期快照替换必须事务化，删除和写入不能分离提交。
- 空记录、日期不一致或解析失败直接拒绝替换。
- 推广页无法确认目标 `mallId` 时拒绝入库。
- 关闭另一个推广页失败时给出错误，避免继续在不确定页面上采集。
- 失败标记只有在明确完成全部请求日期后删除。
- 下载中心未找到业务层时继续等待或按现有超时退出，不通过操作通知层绕过。

## 测试与验证

### 自动测试

1. 同日先写三条、再替换两条，数据库最终仅保留两条。
2. 替换一个日期不影响其他日期。
3. 空快照和日期不一致快照不会删除旧数据。
4. 完整成功删除旧失败标记；失败、零数据和提前终止保留标记。
5. 商品和店铺日期选择器两次点击都使用最后一个可见目标月份面板。
6. 两次日期点击都支持已选中日期单元格。
7. 两个推广候选页中保留 `338884784`，关闭另一个。
8. 没有目标 `mallId` 时不调用推广费入库。
9. 下载中心查询、表格、任务和按钮查找都限定在同一个业务根层。
10. 隐藏旧根层、多个 AG-Grid 或高层通知同时存在时，仍只选择当前可见且结构正确的下载业务层。
11. Python 解析器精确过滤已知样式警告并关闭 workbook。

### 命令验证

- 运行相关聚焦测试；
- 运行完整 Node 测试套件；
- 对修改的 `.mjs` 文件运行 `node --check`；
- 运行 `git diff --check`。

### 真实页面验证

通过现有 CDP Chrome：

1. 保留当前堆叠的“我知道了”通知，不做预清理；先记录可见 AG-Grid 的祖先链、通知层和候选按钮，再确认脚本只通过下载业务层完成查询、任务识别和下载；
2. 对一个商品日期和一个店铺日期完成单日选择并校验输入框；
3. 同时打开两个推广页，确认保留目标店铺页、关闭另一个，并将数据映射到“吃了会快乐零食店”；
4. 重新采集一个已有旧行的日期，核对 JSON 与 SQLite 的 `(shop_name, product_id)` 集合完全一致。

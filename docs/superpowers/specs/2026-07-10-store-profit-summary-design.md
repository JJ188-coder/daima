# 商品报表店铺利润汇总设计

## 目标

在拼多多“商品报表”弹窗的现有绿色利润列后，增加当前店铺、当前报表日期范围的汇总列。

汇总只使用商品报表中已经显示出慧经营绿色利润数据的商品 ID。没有卖出、未被慧经营导出、因此当前绿色列显示 `--` 的商品，不进入 ID 集合、不参与汇总，也不会向数据库发起逐个查询。

店铺名称不参与匹配。唯一匹配条件是：商品报表行的 `itemId` / `goodsId` 与已加载慧经营记录的 `productId` 完全相同。

## 已确认的业务规则

- 拼多多商品报表一次只对应当前登录店铺。
- 一个店铺可能有一页或多页商品。为了不遗漏后面页里有利润的商品，仍需读取每一页的报表行；但每行只做内存里的 ID 是否命中判断，不查询未命中的 ID。
- “可汇总商品”是指 `huiceMap[商品ID]` 存在且 `netProfit` 不为空的商品，也就是当前商品级绿色净利润、净利率等列能够显示数据的商品。
- 慧经营原始净利与调整后净利必须分开保存和展示。
- 调整后净利已经按既有规则入库：`慧经营原始净利 - 1.15 * 订单数 - 销售额 * 2%`。
- 当前业务口径中销售件数等于订单数。
- 多日汇总后的净利率必须重新计算为“汇总调整后净利 / 汇总销售额”，不能相加或平均每日净利率。
- 某个商品的调整后净利润小于零时，该商品所在整行必须标红，不只标红“净利润”单元格。
- 无匹配、未回采、分页未读完的数据必须显示为空，不能补零或估算。

## 数据来源和边界

慧经营的唯一生产数据来源仍是 `tools/huice-export-cdp.mjs`：它在商品分析页选择日期后点“导出全部”，下载完整 XLSX 并将每个商品每日数据写入 `product_profit`。日常定时任务 `scripts/huice-daily.sh` 已经调用这条路径。

商品报表打开时沿用已有 `getHuiceDataByDateRange(start, end)`：它优先从本地 HTTP 服务读取 SQLite 中这一日期范围的慧经营商品记录，服务不可用时再读扩展 storage。返回的记录已经按商品 ID 聚合多日数据。

本功能不新增“把全部拼多多商品 ID 传给数据库”的接口，也不在打开商品报表时重新登录或读取慧经营网页。

## 数据流

```text
慧经营商品分析“导出全部”
  -> XLSX 解析
  -> product_profit（每商品、每日期）
  -> 既有 getHuiceDataByDateRange()
  -> huiceMap: productId -> 已聚合的慧经营记录

拼多多商品报表弹窗（逐页读取）
  -> 提取当前页 itemId / goodsId
  -> 仅当 huiceMap[id].netProfit 不为空时，收录该 ID 和对应记录
  -> 合并所有页的收录记录（按 ID 去重）
  -> 汇总店铺金额并回填绿色店铺列
```

这里的“逐页读取”只是在拼多多页面内查看表格行，目的是找到已有慧经营数据的商品。未卖商品最多经历一次 `Map` 查询，不会产生数据库查询、HTTP 请求或慧经营访问。

读取开始时记录用户原本所在页；结束、超时或失败后都恢复该页。任一页读取失败时，丢弃已收集的临时结果，店铺汇总保持 `--`，绝不展示不完整总数。

## 汇总算法

输入为当前日期范围内的慧经营映射 `huiceMap` 与所有商品报表页收录到的记录集合 `R`。

1. 对每一页每一行读取 `id = String(row.itemId || row.goodsId || '')`。
2. 读取 `record = huiceMap[id]`。仅当 `record.netProfit` 是有效数值时，把 `record` 以 `id` 为键写入集合 `R`。
3. 使用 `R` 中每个商品的一条多日聚合记录分别求和：销售额、慧经营原始净利、包装人工、平台费、调整后净利。
4. 重算调整后净利率：`调整后净利 / 销售额`；销售额为零时返回空值。
5. 计算覆盖：`R.size / 已扫描的商品报表去重 ID 数`。

当前报表中的未卖商品、缺少慧经营记录的商品、以及慧经营记录中利润为空的商品都不进入金额合计。

返回字段：

```json
{
  "scannedProductCount": 245,
  "matchedProductCount": 238,
  "unmatchedProductCount": 7,
  "summary": {
    "salesAmount": 3913.1,
    "rawNetProfit": 1387.8,
    "orderFixedCost": 273.7,
    "platformFee": 78.26,
    "netProfit": 1035.84,
    "netProfitRate": 0.2647
  }
}
```

## 商品报表界面

保留现有 6 个绿色商品级列：

`净利润 | 净利率 | 毛利率 | 退款额 | 推广费比 | 保本ROI`

紧接“保本ROI”、在“趋势”之前新增以下绿色店铺汇总列：

`店铺销售额 | 店铺原始净利 | 包装人工 | 平台费 | 店铺调整净利 | 店铺调整净利率 | 覆盖商品`

店铺汇总属于当前店铺与当前日期范围，因此同一页商品行会显示同一组店铺汇总数字。商品级六列仍然只显示各自商品的数据。

显示规则：

- 金额：`¥1,234.56`；店铺调整净利为负时显示红色。
- 比例：`12.34%`。
- 覆盖：`238 / 245`，前者是已有绿色慧经营利润的商品数，后者是已扫描的报表商品数。
- 商品行的调整后净利润小于零时，该行所有数据单元格都显示红色，包括拼多多原始列、六个商品级绿色列和七个店铺汇总列。店铺调整净利为负但当前商品不亏损时，只标红该店铺调整净利单元格。
- 慧经营未回采、任一拼多多页读取失败、或当前日期没有可用绿色利润商品时，店铺汇总列显示 `--`；浏览器控制台记录明确原因。

## 失败处理与性能

- 不按商品 ID 向数据库逐条或批量搜索；现有日期范围数据只加载一次，后续匹配全部为内存 `Map` 查找。
- 分页读取有单次页切换超时和 200 页保护上限；任何失败都恢复用户原页，不显示部分汇总。
- 对相同“日期范围 + 当前商品报表”缓存一次完成结果；日期、搜索、筛选或报表重新渲染后失效。
- 同一时刻只允许一个分页读取任务，避免 `MutationObserver` 重复翻页。
- 不新增云端服务、不传出慧经营数据；所有数据继续在 `127.0.0.1`、扩展 storage 和已打开的拼多多报表内完成。

## 非目标

- 不按慧经营店铺名称匹配，也不维护手工店铺映射。
- 不查询或汇总报表中没有绿色慧经营利润数据的商品。
- 不改变商品级利润列和既有利润公式。
- 不回写慧经营、拼多多或任何外部系统。
- 不将未匹配商品的利润猜为零。

## 验收标准

1. 一页店铺与多页店铺都能得到完整汇总，且读取结束后页面恢复到用户最初页。
2. 只把当前商品级绿色利润列有数据的商品纳入店铺汇总；未卖商品不导致新的数据库查询。
3. 用所有已收录商品的商品级调整后净利手工相加，等于“店铺调整净利”。
4. 日期切换后，汇总数值与覆盖数同步变化；多日净利率为汇总后重算值。
5. 删除一个本地商品利润记录后，该商品的绿色列变为 `--`，覆盖数减少，汇总不补零。
6. 任一调整后净利润为负的商品行，其所有数据单元格均为红色；相邻盈利商品行保持原有颜色。
7. 慧经营数据不存在或分页读取不完整时，店铺列不显示数值。

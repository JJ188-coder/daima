# 商品报表店铺利润汇总设计

## 目标

在拼多多“商品报表”弹窗的现有绿色利润列后，增加当前店铺、当前报表日期范围的汇总列。汇总对象由商品报表中的完整商品 ID 集合决定，不使用店铺名称做匹配。

这让店名改名、别名或慧经营显示格式差异不再影响结果：只要拼多多报表中的商品 ID 与慧经营导出的商品 ID 相同，该商品就属于本次店铺汇总。

## 已确认的业务规则

- 拼多多商品报表一次只对应当前登录店铺。
- 一个店铺可能只有一页商品，也可能有很多页；汇总必须读取当前报表的全部分页商品 ID。
- 慧经营原始净利与调整后净利必须分开保存和展示。
- 调整后净利已经按既有规则入库：`慧经营原始净利 - 1.15 * 订单数 - 销售额 * 2%`。
- 当前业务口径中销售件数等于订单数。
- 多日汇总后的净利率必须重新计算为“汇总调整后净利 / 汇总销售额”，不能相加或平均每日净利率。
- 无匹配或未回采的数据必须显示为空，不能补零或估算。

## 数据来源和边界

慧经营的唯一生产数据来源仍是 `tools/huice-export-cdp.mjs`：它在商品分析页选择日期后点“导出全部”，下载完整 XLSX 并将每个商品每日数据写入 `product_profit`。日常定时任务 `scripts/huice-daily.sh` 已经调用这条路径。

旧的 `scripts/huice-sync.mjs` 直接读取慧经营当前渲染页，无法保证覆盖慧经营分页；它不作为店铺汇总的数据来源。

店铺汇总不在打开商品报表时重新登录或读取慧经营网页，只读取本机 SQLite / 扩展 storage 中已经导入的数据。因此，用户当前浏览拼多多报表不会被慧经营自动化打断。

## 数据流

```text
慧经营商品分析“导出全部”
  -> XLSX 解析
  -> product_profit（每商品、每日期）

拼多多商品报表弹窗
  -> 逐页读取当前报表全部商品 ID
  -> POST /huice/shop-summary { start, end, productIds }
  -> 按日期和商品 ID 查询 product_profit
  -> 按商品聚合多日记录，再计算店铺汇总
  -> 回填商品报表中的店铺汇总列
```

拼多多页面侧在逐页读取期间记录用户原本所在页。读取完成或失败后都恢复该页。读取期间的 DOM 更新不会触发并发的第二次收集。

## 汇总算法

输入为当前商品报表去重后的商品 ID 集合 `P` 与日期范围 `[start, end]`。

1. 从 `product_profit` 取 `date BETWEEN start AND end AND product_id IN P` 的记录。
2. 先按商品 ID 聚合多日记录，保留已入库的销售额、原始净利、包装人工、平台费和调整后净利。
3. 只将“调整后净利不为空”的商品计入汇总和覆盖数；缺少利润数据的商品不按零处理。
4. 对纳入汇总的商品分别求和：销售额、原始净利、包装人工、平台费、调整后净利。
5. 重新计算调整后净利率：`调整后净利 / 销售额`；销售额为零时返回空值。

返回字段：

```json
{
  "start": "2026-07-02",
  "end": "2026-07-08",
  "productCount": 245,
  "matchedProductCount": 238,
  "unmatchedProductCount": 7,
  "summary": {
    "salesAmount": 3913.1,
    "rawNetProfit": 1387.8,
    "orderFixedCost": 273.7,
    "platformFee": 78.26,
    "netProfit": 1035.84,
    "netProfitRate": 0.2647
  }
}
```

`productCount` 永远是当前拼多多报表中的商品数，`matchedProductCount` 是有可用利润记录的商品数。所有金额和比例只基于匹配商品计算。

## HTTP 接口

新增本地接口：

```text
POST http://127.0.0.1:9911/huice/shop-summary
Content-Type: application/json

{
  "start": "YYYY-MM-DD",
  "end": "YYYY-MM-DD",
  "productIds": ["959800396594", "715688768820"]
}
```

使用 POST 避免多页店铺的大量商品 ID 超出 URL 长度。接口只返回汇总与覆盖计数，不回传整店所有商品明细。服务端接受至多 10,000 个去重后的非空 ID，超过上限返回 400 并显示“不计算”，而不是截断商品列表。

本地服务不可用时，扩展退回到已有的 `chrome.storage.local` 慧经营记录，使用同一套 ID 过滤和汇总规则；两个通道都不可用时显示空值。

## 商品报表界面

保留现有 6 个绿色商品级列：

`净利润 | 净利率 | 毛利率 | 退款额 | 推广费比 | 保本ROI`

紧接“保本ROI”、在“趋势”之前新增以下绿色店铺汇总列：

`店铺销售额 | 店铺原始净利 | 包装人工 | 平台费 | 店铺调整净利 | 店铺调整净利率 | 覆盖商品`

店铺汇总属于当前店铺与当前日期范围，因此同一页商品行会显示同一组店铺汇总数字。这样符合用户要求的位置，也不会遮挡或替代商品自己的利润列。

显示规则：

- 金额：`¥1,234.56`；店铺调整净利为负时显示红色。
- 比例：`12.34%`。
- 覆盖：`238 / 245`。
- 汇总尚在读取全部页、慧经营未回采、HTTP / storage 均无数据、或拼多多分页读取失败时，所有店铺汇总列显示 `--`；浏览器控制台记录明确原因。

## 失败处理与性能

- 商品报表分页读取有单次页切换超时和总页数上限；任何失败都恢复用户原页，不显示部分汇总。
- 汇总请求有 10 秒超时；超时后尝试 storage，仍失败则显示空值。
- 对相同“日期范围 + 当前商品 ID 集合”缓存一次完成结果；日期、搜索、筛选或报表重新渲染后失效。
- 同一时刻只允许一个店铺 ID 收集任务，避免 `MutationObserver` 重复翻页。
- 不新增云端服务、不传出慧经营数据；所有查询继续在 `127.0.0.1` 和扩展本地 storage 内完成。

## 非目标

- 不按慧经营店铺名称匹配，也不维护手工店铺映射。
- 不改变商品级利润列和既有利润公式。
- 不回写慧经营、拼多多或任何外部系统。
- 不将未匹配商品的利润猜为零。

## 验收标准

1. 一页店铺与多页店铺都能得到完整汇总，且读取结束后页面恢复到用户最初页。
2. 用当前报表所有商品行的匹配利润手工相加，等于“店铺调整净利”。
3. 日期切换后，汇总数值与覆盖数同步变化；多日净利率为汇总后重算值。
4. 删除一个本地商品利润记录后，覆盖数减少，汇总不补零。
5. 慧经营数据不存在、接口不可用或分页读取不完整时，店铺列不显示数值。
